# Knox_Dolphin — 결정 레코드 (Decision Log)

> 이 문서는 Knox_Dolphin이 *왜* 지금처럼 설계됐는지의 기억입니다.
> Claude Code는 매 세션 이 문서를 읽고, 이 결정들을 지켜야 합니다.
> 새 결정은 아래에 같은 형식으로 추가하고, 뒤집힌 결정은 삭제하지 말고 `상태: 대체됨`으로 표시합니다.

---

## D1 — 데이터 원천 = Claude Code 대화 로그
- **결정:** 코드 diff 자동 추론이 아니라, 개발자와 AI 코딩 어시스턴트의 대화 로그에서 결정을 끌어낸다.
- **이유:** 결정의 '왜'는 diff에 없고 대화에 명시돼 있다. 따라서 추론이 아니라 추출로 처리 가능.
- **대안:** 커밋 diff 자동 추론.
- **기각 이유:** diff엔 '왜'가 없어 AI가 이유를 추측하게 되고, 지어낸 기억이 자산을 오염시킨다.
- **영향:** 신뢰가 구조적으로 확보됨(추측이 아닌 발췌).
- **상태:** 확정

## D2 — MVP 캡처 방식
- **결정:** 수동 트리거("이 세션 정리해줘") + 추출된 결정만 저장 + 원문 트랜스크립트 영구 비저장.
- **이유:** 추출 품질을 사람이 먼저 검증해야 신뢰가 유지되고, 원문 비저장으로 프라이버시 표면을 최소화.
- **대안:** 실시간 자동 감지 / 전체 트랜스크립트 저장.
- **기각 이유:** 실시간은 복잡하고 결정 뒤집힘에 취약; 전체 저장은 보안 부채.
- **영향:** 추출 로직은 트리거 독립적으로 설계 → 자동화 전환이 쉬움.
- **상태:** 확정

## D3 — 신뢰가 최우선 가치 (불변식)
- **결정:** 신뢰를 제품의 최우선 가치로 삼고, 아래 6가지 불변식을 강제한다.
  1. 출처 없으면 결정 없음 (`source_quote` 필수)
  2. 확인 게이트는 decisions의 *유일한* 쓰기 경로 (사람 승인만 통과)
  3. 회상은 검색된 레코드 안에서만 답하고, 없으면 "기록 없음"이라 답한다 (무근거 답변 금지)
  4. 추출 ≠ 추론 (명시된 것만, 없으면 null)
  5. 원문은 영구 저장하지 않는다
  6. 상태(candidate/confirmed/superseded)는 항상 사용자에게 보인다; 대체된 결정은 삭제하지 않고 표시
- **영향:** 불변식은 `core/trust`에 테스트된 함수로 모아 코드·테스트가 강제하게 한다.
- **상태:** 확정

## D4 — 데이터 모델: 성역 분리
- **결정:** 추출 후보(`candidates`)와 확정 결정(`decisions`)을 별도 테이블로 둔다. 사람이 확인한 것만 `decisions`에 승격.
- **이유:** 확정 테이블을 '성역'으로 지켜 미확정 노이즈가 자산을 오염시키지 않게.
- **대안:** 단일 테이블 + status 컬럼.
- **기각 이유:** 신뢰를 위해 분리가 더 안전.
- **상태:** 확정

## D5 — 프롬프트는 경험적으로 검증
- **결정:** 추출·회상 프롬프트를 종이에서 과설계하지 않고, 실제 로그에 돌려 나온 결과로 개선한다.
- **이유:** 프롬프트 품질은 읽어서 알 수 없고 돌려봐야 안다. 실제 실행에서 '참조 채택' 같은 결함이 드러났다.
- **영향:** few-shot 예시(빈 결과/null/참조 채택)를 추가해 v1 확정. 정밀도 우선(놓침은 복구 가능, 지어냄은 오염).
- **상태:** 확정

## D6 — 로컬 우선 배포  *(신규)*
- **결정:** 호스팅 웹앱이 아니라, 사용자의 컴퓨터에서 도는 **로컬 도구**로 만든다. (localhost에서 도는 Next.js)
- **이유:** 데이터 원천이 로컬 로그이고, 신뢰가 최우선이다. 로컬 우선이면 원문이 기기를 떠나지 않아 프라이버시 약속이 문자 그대로 참이 된다. 서버가 없으니 멀티테넌시·운영 부담·법적 무게도 없다.
- **대안:** 로컬 에이전트+호스팅 백엔드 / 사용자가 로그를 웹에 업로드.
- **기각 이유:** 업로드 방식은 원문이 서버에 닿아 신뢰 원칙과 충돌. 호스팅은 검증 전 과도한 비용·책임.
- **영향:** 인증·DB서버가 MVP에서 빠짐(아래 D7, D8). 추출 LLM 비용은 사용자 본인 API 키로 부담 가능. 호스팅(레벨 2)은 검증된 뒤에만.
- **상태:** 확정

## D7 — SQLite + 벡터 확장  *(D4의 저장소 선택을 대체)*
- **결정:** Postgres + pgvector 대신 **SQLite + 벡터 검색 확장**(예: sqlite-vec)을 쓴다. (정확한 확장은 Claude Code가 선택)
- **이유:** 로컬 배포에선 SQLite가 '파일 하나'라 설치할 DB 서버가 없다. 비기술 사용자 마찰 최소화.
- **대체 대상:** "PostgreSQL + pgvector" (호스팅 전제로 내렸던 선택)
- **영향:** 테이블 4개 구조(users 제외 가능)·신뢰 불변식은 그대로, 타입·인덱스만 SQLite식으로.
- **상태:** 확정 (supersedes pgvector)

## D8 — 로컬 MVP는 인증 없음  *(PRD '#1 GitHub 인증'을 연기)*
- **결정:** 레벨 0/1 로컬 MVP에는 로그인을 두지 않는다. 내 기기, 내 데이터.
- **이유:** 로컬 단일 사용자라 인증이 불필요. 복잡도 제거.
- **영향:** `users` 테이블·OAuth 화면 불필요. GitHub OAuth는 호스팅(레벨 2)에서 부활.
- **상태:** 확정 (PRD MVP #1을 레벨 2로 연기)

## D9 — 벡터 확장은 회상(6단계)에서 붙인다  *(ARCHITECTURE 빌드순서 2단계 일부를 연기)*
- **결정:** 스키마 마이그레이션(2단계)에서는 `decisions.embedding`을 BLOB 컬럼으로만 두고, 실제 SQLite 벡터 확장(sqlite-vec 등) 로딩은 회상(6단계)에서 붙인다.
- **이유:** 임베딩은 승격 시점에만 생성되므로(D7) 회상 전엔 벡터 검색이 필요 없다. 반면 네이티브 확장을 `node:sqlite`에 로딩하는 것이 Windows에서 가장 깨지기 쉬운 조각이라, 실제로 필요한 시점에 붙이는 게 2단계를 깨끗하고 테스트 가능한 슬라이스로 유지한다.
- **대안:** 2단계에서 확장까지 한 번에 로딩.
- **기각 이유:** 가장 위험한 의존성을 아직 쓰지도 않는 시점에 끌어와 슬라이스를 오염시킨다.
- **영향:** 테이블 구조·불변식은 그대로. 6단계에서 임베딩 생성 + 벡터 인덱스를 추가한다.
- **상태:** 확정

## D10 — 확인 게이트는 웹 UI 전용이 아니다: 대화 내 승인 허용  *(불변식 ② 재해석, 약화 아님)*
- **결정:** 결정 승격(candidates→decisions)을 웹 UI 버튼뿐 아니라 **MCP 도구(`knox_confirm_candidate`)로 대화 안에서**도 할 수 있게 한다. Claude가 후보를 사람에게 보여주고 명시적 승인을 받은 뒤에만 도구를 호출한다.
- **이유:** 불변식 ②의 본질은 "**사람 승인**이 유일한 쓰기"이지 "웹 UI가 유일한 쓰기"가 아니다. 플러그인 사용 흐름에서 웹앱을 띄우지 않고도 승인할 수 있어야 실용적이다.
- **하드 게이트:** Claude의 정직성이 아니라 **Claude Code의 도구 권한 프롬프트**가 사람의 클릭을 강제한다. Claude는 혼자 승인할 수 없다.
- **트레이드오프:** 웹 UI 승인(후보 전문을 직접 보고 클릭)이 가장 강하고, 대화 승인은 Claude가 후보를 충실히 보여준다는 신뢰가 약간 더 든다. 사용자가 이 도구를 "항상 허용"으로 두면 per-decision 게이트가 사라지므로 권하지 않는다.
- **영향:** 웹앱은 조회 중심이 되고, 승인은 대화에서 가능. `decisions`의 유일한 writer는 여전히 `core/confirmation`(웹·MCP 둘 다 이 경로를 거침).
- **상태:** 확정
